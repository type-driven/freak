# Plan: Update plugin-effect consumer + comprehensive tests

## Goal
Update `effectPlugin()` to use the new per-app `setEffectRunner(app, runner)` API instead of the removed global `setEffectResolver()`, and write tests that verify all 4 Phase 6 success criteria: per-app isolation, `app.get()` Effect dispatch, `app.use()` Effect middleware, and existing tests unchanged.

## Context

@.planning/PROJECT.md
@.planning/ROADMAP.md
@.planning/phases/06-fresh-core-plumbing/06-RESEARCH.md
@.planning/phases/06-fresh-core-plumbing/06-01-PLAN.md

**After Plan 01:**
- `setEffectResolver` no longer exists in `internals.ts`. It has been replaced by `setEffectRunner(app, runner)` and `getEffectRunner(app)`.
- `isEffectLike` is now exported from `@fresh/core/internal` (via `internals.ts`).
- `runMiddlewares` resolves Effect values when an effectRunner is configured on the app.
- `renderRoute` takes an effectRunner parameter instead of reading a global.

**The problem with effectPlugin():**
Currently `effectPlugin()` calls `setEffectResolver(resolver)` which sets a global. It needs to change to call `setEffectRunner(app, resolver)`. But `effectPlugin()` returns a middleware function — it doesn't receive the `App` instance. The plugin needs to be restructured so that it gets the app reference.

**Design decision:** The cleanest approach is to change `effectPlugin()` to accept the `App` instance as a parameter. Since `effectPlugin()` is called like `app.use(effectPlugin())`, we change it to `app.use(effectPlugin(app))` — or better, `app.use(effectPlugin(app, { layer: ... }))`. The app reference is the first argument.

Alternatively, we could make `effectPlugin` return a function that is called with the app, but that breaks the existing `app.use(effectPlugin())` pattern too much. The simplest migration path: `effectPlugin(app)` or `effectPlugin(app, options)`.

**Important:** The existing v1 API `effectPlugin({ layer })` must eventually be maintained via a compat shim (Phase 10). For Phase 6, the signature changes to require the app. The v1 tests need to be updated to pass the app.

Create **packages/plugin-effect/tests/per_app_test.ts** with tests for all 4 Phase 6 success criteria:

**SC-1: Per-app isolation (CORE-01)**

```typescript
import { assertEquals } from "jsr:@std/assert@1";
import { Effect, Layer, ServiceMap } from "effect";
import { App } from "@fresh/core";
import { FakeServer } from "../../fresh/src/test_utils.ts";
import { effectPlugin } from "../src/mod.ts";

// Two services with different values
const ServiceA = ServiceMap.Service<{ value: string }>("ServiceA");
const ServiceB = ServiceMap.Service<{ value: string }>("ServiceB");

const LayerA = Layer.succeed(ServiceA, { value: "from-app-A" });
const LayerB = Layer.succeed(ServiceB, { value: "from-app-B" });

Deno.test("per-app isolation: two apps each see only their own services", async () => {
  const appA = new App();
  appA.use(effectPlugin(appA, { layer: LayerA }));
  appA.route("/", {
    handler: () => Effect.gen(function* () {
      const svc = yield* ServiceA;
      return new Response(svc.value);
    }),
  });

  const appB = new App();
  appB.use(effectPlugin(appB, { layer: LayerB }));
  appB.route("/", {
    handler: () => Effect.gen(function* () {
      const svc = yield* ServiceB;
      return new Response(svc.value);
    }),
  });

  const serverA = new FakeServer(appA.handler());
  const serverB = new FakeServer(appB.handler());

  const resA = await serverA.get("/");
  const resB = await serverB.get("/");

  assertEquals(await resA.text(), "from-app-A");
  assertEquals(await resB.text(), "from-app-B");
});
```

**SC-3: Effect middleware via app.use() (CORE-03)**

```typescript
Deno.test("app.use() middleware returning Effect is resolved", async () => {
  const app = new App();
  app.use(effectPlugin(app));

  // Effect middleware that injects a value into ctx.state
  app.use((ctx) => {
    return Effect.gen(function* () {
      (ctx.state as Record<string, unknown>).injected = "from-effect-middleware";
      return ctx.next();
    }).pipe(Effect.flatten);
  });

  // Downstream handler reads the injected value
  app.get("/check", (ctx) => {
    const injected = (ctx.state as Record<string, unknown>).injected;
    return new Response(String(injected));
  });

  const server = new FakeServer(app.handler());
  const res = await server.get("/check");

  assertEquals(res.status, 200);
  assertEquals(await res.text(), "from-effect-middleware");
});
```

Note on the middleware Effect test: The middleware returns an `Effect<Response>`. The middleware sets `ctx.state.injected`, then returns `ctx.next()` wrapped in Effect.gen + Effect.flatten. The `runMiddlewares` Effect resolution will call the effectRunner on this Effect, which runs it through the ManagedRuntime, producing the Response from `ctx.next()`.

Actually, there is a subtlety: `ctx.next()` returns `Promise<Response>`, and wrapping it in `Effect.gen` with `yield*` won't work directly on Promises in Effect v4. Use `Effect.promise` instead:

```typescript
app.use((ctx) => {
  return Effect.gen(function* () {
    (ctx.state as Record<string, unknown>).injected = "from-effect-middleware";
    const response = yield* Effect.promise(() => ctx.next());
    return response;
  });
});
```

If `Effect.promise` is not the right API in v4, use `Effect.tryPromise`. Check the effect-smol API. The key pattern is: the middleware returns an `Effect<Response>`, the Effect sets state and delegates to `ctx.next()`.

**Step A: Add `initAtomHydration()` to island.ts**

In `packages/plugin-effect/src/island.ts`, add the following function AFTER the
existing registry creation and BEFORE the hook functions:

```typescript
// Tracks which atom keys were pre-seeded for orphan detection
const _hydratedKeys = new Set<string>();

/**
 * Initialize atom hydration from server-serialized state.
 * Must be called BEFORE boot() in the generated client script.
 *
 * Reads the __FRSH_ATOM_STATE script tag, parses the JSON, and calls
 * registry.setSerializable(key, encoded) for each atom. The registry
 * will decode the value lazily on first get() via the atom's schema.
 *
 * @param serialized - JSON string of { [atomKey]: encodedValue } pairs.
 *   If not provided, reads from the __FRSH_ATOM_STATE script tag in the DOM.
 */
export function initAtomHydration(serialized?: string): void {
  let json: string | null | undefined = serialized;
  if (json === undefined) {
    const el = globalThis.document?.getElementById("__FRSH_ATOM_STATE");
    if (!el) return;
    json = el.textContent;
  }
  if (!json) return;

  let data: Record<string, unknown>;
  try {
    data = JSON.parse(json);
  } catch {
    // Malformed hydration data -- silently skip rather than crash boot
    if (typeof globalThis.console !== "undefined") {
      console.warn("[fresh-effect] Malformed atom hydration data, skipping.");
    }
    return;
  }

  for (const [key, encoded] of Object.entries(data)) {
    registry.setSerializable(key, encoded);
    _hydratedKeys.add(key);
  }
}

/**
 * Check for orphaned atom keys -- keys that were hydrated from the server
 * but never consumed by any useAtomValue/useAtom call.
 * Called after boot/render completes. Dev-only warning.
 */
export function _checkOrphanedKeys(): void {
  // This is a best-effort check. We cannot know exactly which keys were
  // consumed by the registry without instrumenting get(). For now, we
  // track hydrated keys and let the user correlate.
  // A more robust approach would check registry internals, but that
  // couples to AtomRegistry private state.
  if (_hydratedKeys.size > 0 && typeof globalThis.console !== "undefined") {
    // Schedule after current task to let boot() complete
    setTimeout(() => {
      // If all keys were consumed, the atoms were created and ensureNode
      // was called. We can't easily check this without registry internals,
      // so we skip the orphan warning for now -- it would require
      // instrumenting registry.get() which is not worth the complexity.
      // The orphan warning is a nice-to-have; the core hydration works
      // without it.
    }, 0);
  }
}
```

Note: The orphan key detection is genuinely difficult without instrumenting
`registry.get()` internals. The CONTEXT.md says "Dev warning for orphaned keys"
as a locked decision. Implement a pragmatic version: After `initAtomHydration`,
store the set of hydrated keys. Schedule a check via `setTimeout` after boot
completes. At that point, query the registry for each hydrated key's atom -- if
`registry.getNodes()` (if available) does not contain a node for that key, warn.
Check the actual AtomRegistry API for a `getNodes()` or equivalent method. If
no such API exists, document that orphan detection requires future AtomRegistry
instrumentation and skip the active check for now, keeping the infrastructure
(`_hydratedKeys` set) in place.

**Step B: Modify FreshRuntimeScript to inject initAtomHydration call in boot script**

In `packages/fresh/src/runtime/server/preact_hooks.ts`, in the `FreshRuntimeScript`
function, in the non-partial `else` branch where `scriptContent` is built:

The current code builds:
```javascript
`import { boot } from "${basePath}${runtimeUrl}";${islandImports}boot(${islandObj},${serializedProps});`
```

When atom hydration data exists (atomJson is non-null from Plan 01's hook), the
script content must be modified to ALSO import `initAtomHydration` and call it
before `boot()`.

However, there is a challenge: the boot script needs to know the URL path to the
plugin-effect island module. This path must come from the build cache (like island
files). We need to handle this carefully.

**Approach:** Use a second hook pattern. Add a `getAtomHydrationIslandUrl` getter
alongside the existing atom hydration hook. The plugin registers the URL to its
island entry point at setup time.

In `packages/fresh/src/segments.ts`, add:

```typescript
let _atomHydrationBootScript: string | null = null;

export function setAtomHydrationBootScript(script: string): void {
  _atomHydrationBootScript = script;
}

export function getAtomHydrationBootScript(): string | null {
  return _atomHydrationBootScript;
}
```

Export `setAtomHydrationBootScript` from `packages/fresh/src/internals.ts`.

In `packages/plugin-effect/src/mod.ts`, register the boot script snippet:

```typescript
import { setAtomHydrationBootScript } from "@fresh/core/internal";

// In effectPlugin(), after registering the hydration hook:
// The boot script snippet that FreshRuntimeScript will inject before boot().
// Uses document.getElementById to read the JSON data tag.
setAtomHydrationBootScript(
  `import{initAtomHydration as __iAH}from"@fresh/plugin-effect/island";` +
  `const __aS=document.getElementById("__FRSH_ATOM_STATE");` +
  `if(__aS)__iAH(__aS.textContent);`
);
```

Wait -- this approach has a problem. The client import `@fresh/plugin-effect/island`
won't resolve in the browser. The client needs a resolvable URL, not a bare specifier.

**Better approach:** Instead of importing from plugin-effect in the boot script,
inline the hydration logic directly in the boot script. The logic is small (3 lines):

```javascript
const __aS=document.getElementById("__FRSH_ATOM_STATE");
if(__aS){const __d=JSON.parse(__aS.textContent);...}
```

But this requires knowing the registry.setSerializable API which lives in island.ts.

**Simplest correct approach:** The `initAtomHydration` function in island.ts should
read from the DOM itself (it already does in the code above when no argument is
passed). Then the boot script just needs to call `initAtomHydration()` with no args.
BUT the boot script must import it -- and the import URL must be resolvable.

Actually, looking at how Fresh handles islands: island files are registered in the
build cache and get resolvable URLs. The `plugin-effect/island.ts` is exported as
`"./island"` in deno.json. During build, the build system processes island imports.

**Most pragmatic approach for now:** Rather than injecting into the boot script
(which requires build system integration to resolve the island module URL), use a
SEPARATE inline script tag that runs before the boot script. The script tag:

1. Reads `__FRSH_ATOM_STATE` script tag
2. Stores the data globally (e.g., on `globalThis.__FRSH_ATOM_DATA`)
3. The island.ts `initAtomHydration()` reads from `globalThis.__FRSH_ATOM_DATA`
   OR from the DOM (if the global is not set)

Actually, even simpler: the `initAtomHydration()` function in island.ts already
reads from the DOM when called with no args. The island module is imported by the
user's island component (e.g., `import { useAtom } from "@fresh/plugin-effect/island"`).
When island.ts is imported, we can auto-run hydration at module load time.

**Final approach (simplest, no boot script modification needed):**

In island.ts, at module level (after registry creation), auto-detect and apply
hydration:

```typescript
// Auto-initialize hydration from DOM if available.
// Module-level code runs at import time, which happens BEFORE boot() calls revive().
// boot() schedules rendering via scheduler.postTask or setTimeout(0), so module
// imports complete first.
if (typeof globalThis.document !== "undefined") {
  const el = globalThis.document.getElementById("__FRSH_ATOM_STATE");
  if (el?.textContent) {
    try {
      const data = JSON.parse(el.textContent) as Record<string, unknown>;
      for (const [key, encoded] of Object.entries(data)) {
        registry.setSerializable(key, encoded);
        _hydratedKeys.add(key);
      }
    } catch {
      // Silently skip malformed data
    }
  }
}
```

This is the cleanest approach because:
- No boot script modification needed
- No new Fresh core hook for the boot script
- Module-level code in island.ts runs when ANY island imports `useAtom` etc.
- Import resolution happens before `boot()` calls `revive()` (which uses scheduler)
- The `initAtomHydration()` function is still exported for manual/test use

This removes the need for `setAtomHydrationBootScript` entirely. Only keep the
`setAtomHydrationHook` + `getAtomHydrationHook` from Plan 01 (for the JSON script
tag emission).

So for Step B, there is NO change to FreshRuntimeScript's boot script content
beyond what Plan 01 already did (emitting the `__FRSH_ATOM_STATE` JSON tag).

Revert/remove any `setAtomHydrationBootScript` / `getAtomHydrationBootScript` code
if it was added. The segments.ts and internals.ts only need the hydration hook
(from Plan 01), not a boot script hook.

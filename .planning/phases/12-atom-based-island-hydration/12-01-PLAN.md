The document must contain these sections:

**1. Problem Statement** Restate the Phase 12 question from the roadmap: "Since
Freak uses Effect v4 atoms instead of Preact signals, does the existing
serialization/reviver pipeline have a mismatch? Should atom-based hydration
replace signal hydration, layer on top, or use a new protocol?"

**2. Current Architecture (satisfies SC-1)** Document both hydration channels
with server-side and client-side flow:

Channel 1 — Island Props (Signals):

- Server: route handler -> serialize(props) via stringify.ts -> ENCODED_SIGNAL
  objects -> __FRSH_ISLAND_PROPS in HTML
- Client: boot.ts -> reviver.ts -> reconstructs Signal/Computed objects ->
  islands receive Signal props
- Key files: stringify.ts, reviver.ts, preact_hooks.ts, boot.ts

Channel 2 — Atom State (Effect v4 TxRef):

- Server: setAtom(ctx, atom, value) -> per-request Map on
  ctx.state[ATOM_HYDRATION_KEY] -> serializeAtomHydration() ->
  FreshRuntimeScript emits __FRSH_ATOM_STATE script tag
- Client: island-atoms.ts module-level auto-init -> reads __FRSH_ATOM_STATE ->
  registry.setSerializable(key, encoded) -> useAtomValue() returns hydrated
  value on first render
- Key files: hydration.ts, segments.ts (hook), preact_hooks.ts (emission),
  island-atoms.ts (client)

Explicitly note: the two channels do NOT interact. Atoms never enter the
stringify/reviver pipeline. Signals never enter the __FRSH_ATOM_STATE channel.

**3. Decision (satisfies SC-2)** Decision: Atom-based hydration LAYERS ON TOP of
signal hydration. No replacement, no new protocol.

Rationale:

- Signal hydration handles reactive island props (Preact-idiomatic, zero
  overhead for non-Effect users)
- Atom hydration handles global application state (Effect-idiomatic,
  server-authoritative)
- They serve different purposes with different data shapes
- Replacing signals with atoms would break all non-Effect Fresh users
- A new unified protocol would add complexity with no benefit (the channels are
  orthogonal)

**4. API Surface Assessment (satisfies SC-3)** Document each public API:

- `setAtom(ctx, atom, value)` — stable, no breaking changes needed
- `initAtomHydration(serialized?)` — stable, no breaking changes needed
- `useAtom()`, `useAtomValue()`, `useAtomSet()` — stable, no changes
- `setAtomHydrationHook()` — internal, stable
- `Atom.serializable({ key, schema })` — Effect v4 API, used correctly

Conclusion: No breaking changes. No migration path needed (nothing to migrate
from/to).

**5. Identified Gaps** For each of the four gaps from 12-RESEARCH.md, record the
gap, its severity, and an explicit ACCEPT or DEFER decision:

Gap 1 (Partials don't emit __FRSH_ATOM_STATE): Medium severity. ACCEPT for now —
Partials with atom mutations is an edge case. Document as known limitation. If
needed later, the fix is to emit atom state in the Partial fragment response.

Gap 2 (setAtomHydrationHookForApp stored but unused by FreshRuntimeScript): Low
severity. ACCEPT — FreshRuntimeScript calls getAtomHydrationHook() (global from
segments.ts), not getAtomHydrationHookForApp() (per-app from app.ts). The
per-app hook was added in Phase 06 for future multi-app support but is not wired
into the render path. Single-app is the supported model.

Gap 3 (Global _atomHydrationHook is last-writer-wins): Low severity. ACCEPT —
same conditions as Gap 2. Consistent with the _effectResolver global pattern.
Multi-app atom hydration is not a current requirement.

Gap 4 (hydration.ts duplicated between packages/plugin-effect and
packages/effect): Low severity. DEFER — when @fresh/plugin-effect is formally
deprecated and removed, the duplication resolves itself. Until then, the code is
stable and identical.

**6. Conclusion** The atom-based island hydration system is complete and
architecturally sound. The dual-channel design is correct. No code changes are
recommended for this phase.

Do NOT invent content beyond what the research found. Do NOT propose code
changes. The document header should be:

- Date: 2026-02-28
- Status: Accepted
- Decision: one-line summary (atom-based hydration layers on top of signal
  hydration as a parallel channel)

**Step 3: Implement `httpApi()` method**

Add this method to the `EffectApp` class, after `mountApp()` and before `get()`:

````typescript
/**
 * Mount an Effect HttpApi at the given path prefix.
 *
 * Builds the HttpApi layer with group implementations, converts it to a web
 * handler via `HttpRouter.toWebHandler()`, and registers a Fresh middleware
 * at `prefix` that delegates matching requests to the Effect HTTP stack.
 *
 * The Effect sub-handler shares the app's `ManagedRuntime.memoMap`, so group
 * implementations can access services from the app's Layer if explicitly
 * composed into the group layer.
 *
 * The sub-handler's `dispose()` is called automatically when `EffectApp.dispose()`
 * is invoked.
 *
 * @param prefix - URL path prefix (e.g., "/api"). Requests matching this prefix
 *   are forwarded to the Effect handler. Must start with "/".
 * @param api - The HttpApi definition (created with `HttpApi.make(...).add(...)`)
 * @param groupLayers - One or more Layer values providing group implementations
 *   (created with `HttpApiBuilder.group(api, name, build)`)
 *
 * @example
 * ```typescript
 * const Api = HttpApi.make("myApi").add(
 *   HttpApiGroup.make("users").prefix("/users").add(
 *     HttpApiEndpoint.get("list", "/", { success: Schema.Array(UserSchema) })
 *   )
 * );
 * const UsersLive = HttpApiBuilder.group(Api, "users", (h) =>
 *   h.handle("list", () => Effect.succeed([]))
 * );
 * app.httpApi("/api", Api, UsersLive);
 * ```
 */
// deno-lint-ignore no-explicit-any
httpApi(prefix: string, api: any, ...groupLayers: any[]): this {
  // Build the complete API layer: HttpApiBuilder.layer(api) + group impls + infra services
  const apiLayer = HttpApiBuilder.layer(api).pipe(
    Layer.provide(Layer.mergeAll(...groupLayers)),
    Layer.provide(HttpServer.layerServices),
  );

  // Convert to a web handler. Share the app's memoMap so that service instances
  // built by the main ManagedRuntime can be reused in group implementations
  // (when the user explicitly composes AppLayer into their group layer).
  const { handler, dispose } = HttpRouter.toWebHandler(apiLayer, {
    memoMap: this.#runtime.memoMap,
  });

  // Store disposer for cleanup
  this.#httpApiDisposers.push(dispose);

  // Register a Fresh middleware at the prefix. The Effect handler owns all
  // requests at this prefix -- a 404 from the Effect handler is intentional
  // (the route matched the prefix but the handler returned NotFound).
  this.#app.use(prefix, async (ctx) => {
    return await handler(ctx.req);
  });

  return this;
}
````

**Design decisions and why:**

1. **Explicit `prefix` parameter** (not auto-detected from api.groups):
   Extracting prefixes from HttpApiGroup internals would be fragile and require
   reaching into private fields. An explicit prefix is clear, avoids ambiguity,
   and matches Fresh's `app.use(path, ...)` pattern.

2. **`memoMap` sharing**: `ManagedRuntime` exposes `memoMap` (verified at line
   47 of ManagedRuntime.ts) and `toWebHandler` accepts `options.memoMap`
   (verified at line 1132). This enables sharing of service instances between
   the main runtime and the HttpApi sub-handler without requiring a single
   runtime. Group implementations that compose the app's Layer will get cached
   service instances.

3. **No 404 fallthrough**: Because the middleware is registered at an explicit
   prefix, the Effect handler owns all requests at that prefix. A 404 from the
   Effect handler is intentional (HttpApiError.NotFound) and should be returned
   to the client, not passed to Fresh's routing.

4. **`any` types for api and groupLayers**: The HttpApi generic types
   (HttpApi<Id, Groups>, Layer<HttpApiGroup.ToService<Id, Groups>, ...>) are
   complex and would require importing many types. Using `any` at the boundary
   keeps the API simple while the Effect type system enforces correctness at the
   call site (HttpApiBuilder.group already type-checks against the api
   definition). This matches the pattern used throughout EffectApp's proxy
   methods.

**Step 5: Fix signal disposal path so SIGINT/SIGTERM calls EffectApp.dispose()**

This is CRITICAL. Currently, `registerSignalDisposal` in
`packages/effect/src/runtime.ts` accepts a `ManagedRuntime` and calls
`runtime.dispose()` directly on signal. This bypasses `EffectApp.dispose()`,
which means httpApi sub-handler runtimes will NOT be disposed on SIGINT/SIGTERM.

**Fix in `packages/effect/src/runtime.ts`:**

Change `registerSignalDisposal` to accept a generic dispose function instead of
a `ManagedRuntime`:

```typescript
/**
 * Register a dispose function to run on SIGINT and SIGTERM signals.
 * Returns a cleanup function that removes the signal listeners (for testing).
 *
 * Uses signal handlers instead of the `unload` event -- signals fire reliably
 * on Deno.serve shutdown and are the idiomatic lifecycle mechanism for v2.
 */
export function registerSignalDisposal(
  disposeFn: () => Promise<void>,
): () => void {
  function onSignal(): void {
    Deno.removeSignalListener("SIGINT", onSignal);
    if (Deno.build.os !== "windows") {
      Deno.removeSignalListener("SIGTERM", onSignal);
    }
    void (async () => {
      try {
        await disposeFn();
      } catch (_) {
        // best-effort
      } finally {
        Deno.exit(0);
      }
    })();
  }

  Deno.addSignalListener("SIGINT", onSignal);
  if (Deno.build.os !== "windows") {
    Deno.addSignalListener("SIGTERM", onSignal);
  }

  return () => {
    Deno.removeSignalListener("SIGINT", onSignal);
    if (Deno.build.os !== "windows") {
      Deno.removeSignalListener("SIGTERM", onSignal);
    }
  };
}
```

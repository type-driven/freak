    Create six Deno scripts in `packages/benchmarks/scripts/`. All scripts use `Deno.Command` for subprocess management. All paths should be resolved relative to the scripts directory using `import.meta.dirname` or `new URL("../", import.meta.url)`.

    Define a shared constant for the three apps used across all scripts:

    ```typescript
    const APPS = [
      { name: "upstream", dir: "apps/upstream-app", port: 8003 },
      { name: "freak-plain", dir: "apps/freak-plain-app", port: 8002 },
      { name: "freak-effect", dir: "apps/freak-app", port: 8001 },
    ] as const;
    ```

    **scripts/check_tools.ts:**
    - Export `async function checkTools(): Promise<void>`
    - Check for `oha` and `hyperfine` using `Deno.Command("which", { args: [name] })` with `stdout: "null"`, `stderr: "null"` — check exit code 0
    - If either is missing, print clear install instructions and throw:
      ```
      Missing required tools:
        - oha: brew install oha
        - hyperfine: brew install hyperfine
      ```
    - Export the function so bench.ts can call it

    **scripts/bench_throughput.ts:**
    - Export `async function runThroughputBench(benchRoot: string): Promise<ThroughputResult[]>`
    - Define `ThroughputResult` type: `{ name: string, requestsPerSec: number, latencyP50: number, latencyP90: number, latencyP99: number }`
    - For each app in APPS:
      1. Build the app first: `deno run -A {appDir}/dev.ts build` (cwd: benchRoot)
      2. Start server: `deno serve -A --port={port} _fresh/server.js` (cwd: resolved appDir)
      3. Poll `http://localhost:{port}/` until ready (max 50 attempts, 100ms apart)
      4. Run warmup: `oha -n 2000 -c 10 --no-tui http://localhost:{port}/api/todos` (discard output)
      5. Run measurement: `oha -n 10000 -c 50 --no-tui --output-format json http://localhost:{port}/api/todos`
      6. Parse JSON stdout — extract `summary.requestsPerSec`, `latencyPercentiles.p50`, `.p90`, `.p99` (all in seconds, convert latency to milliseconds for display)
      7. Kill server with SIGTERM, await status
      8. Push result to array
    - Return results array
    - IMPORTANT: Kill any existing processes on the port before starting: `lsof -ti:{port} | xargs kill -9 2>/dev/null; true`

    **scripts/bench_build.ts:**
    - Export `async function runBuildTimeBench(benchRoot: string): Promise<BuildResult[]>`
    - Define `BuildResult` type: `{ name: string, meanSecs: number, stddevSecs: number, runs: number }`
    - Use `hyperfine` to compare all three apps in a single invocation:
      ```
      hyperfine --warmup 1 --runs 5 --export-json /tmp/bench_build.json \
        --prepare 'rm -rf apps/freak-app/_fresh' \
        'deno run -A apps/freak-app/dev.ts build' \
        --prepare 'rm -rf apps/freak-plain-app/_fresh' \
        'deno run -A apps/freak-plain-app/dev.ts build' \
        --prepare 'rm -rf apps/upstream-app/_fresh' \
        'deno run -A apps/upstream-app/dev.ts build'
      ```
      Run with cwd set to benchRoot.
    - Actually, hyperfine with multiple commands and per-command `--prepare` is tricky. Simpler approach: run hyperfine THREE times, once per app, each with its own --prepare and --export-json to a temp file. Parse each temp file.
    - Parse hyperfine JSON: `results[0].mean` (seconds), `results[0].stddev` (seconds)
    - Return array of BuildResult

    **scripts/bench_bundle.ts:**
    - Export `async function runBundleSizeBench(benchRoot: string): Promise<BundleResult[]>`
    - Define `BundleResult` type: `{ name: string, totalRawKB: number, totalGzipKB: number, fileCount: number }`
    - For each app:
      1. Build the app: `deno run -A {appDir}/dev.ts build` (cwd: benchRoot)
      2. Walk `{appDir}/_fresh/static/` recursively
      3. For each `.js` file: read file, sum `file.byteLength` for raw size
      4. For gzip size: use `new CompressionStream("gzip")` on each file's bytes, sum compressed sizes
      5. Report total raw KB, total gzip KB, file count
    - Return array of BundleResult

    **scripts/bench_startup.ts:**
    - Export `async function runStartupBench(benchRoot: string): Promise<StartupResult[]>`
    - Define `StartupResult` type: `{ name: string, meanMs: number, runs: number }`
    - For each app, run 5 iterations:
      1. Build app once before the loop (deno run -A dev.ts build)
      2. Per iteration:
         a. Record `performance.now()` as start
         b. Spawn `deno serve -A --port={port} _fresh/server.js` (cwd: appDir)
         c. Poll `http://localhost:{port}/` every 50ms until 200 response (max 100 attempts = 5s timeout)
         d. Record `performance.now()` as end
         e. Kill server, await status
         f. Store `end - start` in milliseconds
      3. After loop: compute mean of 5 runs
    - Return array of StartupResult
    - Kill any existing process on the port before each iteration

    **scripts/bench.ts:**
    - This is the main orchestrator. Export nothing — runs as a script.
    - Import and call all other scripts:
      ```typescript
      import { checkTools } from "./check_tools.ts";
      import { runThroughputBench } from "./bench_throughput.ts";
      import { runBuildTimeBench } from "./bench_build.ts";
      import { runBundleSizeBench } from "./bench_bundle.ts";
      import { runStartupBench } from "./bench_startup.ts";
      ```
    - Resolve benchRoot: `new URL("../", import.meta.url).pathname`
    - Step 1: `await checkTools()`
    - Step 2: Print "Running throughput benchmark..." then `const throughput = await runThroughputBench(benchRoot)`
    - Step 3: Print "Running build time benchmark..." then `const buildTime = await runBuildTimeBench(benchRoot)`
    - Step 4: Print "Running bundle size benchmark..." then `const bundleSize = await runBundleSizeBench(benchRoot)`
    - Step 5: Print "Running startup time benchmark..." then `const startup = await runStartupBench(benchRoot)`
    - Step 6: Collect system metadata:
      - `Deno.build.arch`, `Deno.build.os`
      - `Deno.version.deno`, `Deno.version.v8`, `Deno.version.typescript`
      - Date: `new Date().toISOString().split("T")[0]`
    - Step 7: Render RESULTS.md using template (see below). Helper function `renderResults(meta, throughput, buildTime, bundleSize, startup): string` that builds the markdown string.
    - Step 8: Write to `{benchRoot}/RESULTS.md`
    - Step 9: Print "Benchmark complete! Results written to packages/benchmarks/RESULTS.md"

  <name>Task 3: Write root-cause analysis for Effect overhead</name>
  <files>
    packages/benchmarks/RESULTS.md

  <action>
    After the benchmark suite completes and RESULTS.md is generated with real numbers, review the Effect overhead data and expand the "Notes on Effect Overhead" section with specific root-cause explanations. This goes beyond the canned threshold-triggered commentary that bench.ts auto-generates.

    Read the generated RESULTS.md and analyze the actual numbers. Then append or replace the "Notes on Effect Overhead" section content with detailed root-cause analysis covering each dimension where Effect adds measurable overhead:

    **Throughput overhead root causes (if measurable):**
    - ManagedRuntime dispatch path: each request goes through `ManagedRuntime.runPromise(effect)` which allocates a Fiber, enters the Effect scheduler, and resolves the result. This adds per-request overhead compared to a plain function call.
    - Fiber scheduling cost: `Effect.gen(function*() { ... })` creates a generator-backed fiber. Each `yield*` is a suspension point that the Effect scheduler must resume. For a trivial handler (return static JSON), this overhead dominates since the actual work is near-zero.
    - Layer service lookup: `yield* TodoService` resolves the service from the pre-built layer context. While the layer is pre-built at startup (not per-request), the Context lookup still has a cost per invocation.
    - Note whether the overhead percentage is acceptable for real-world handlers where actual I/O (database, network) would dominate execution time.

    **Build time overhead root causes (if measurable):**
    - Additional esbuild entries: the Effect npm dependency tree adds modules that esbuild must resolve and bundle, even if tree-shaken.
    - Whether the difference is statistically significant given the stddev.

    **Bundle size overhead root causes (if measurable):**
    - Whether Effect code appears in client bundles (it should NOT for server-only handlers).
    - If it does appear: identify which island import pulls in Effect code.
    - If bundle sizes are identical across all three apps: note that server-only Effect usage has zero client bundle impact.

    **Startup time overhead root causes (if measurable):**
    - Layer construction: `ManagedRuntime.make(layer)` runs at startup, building the service dependency graph. For a trivial TodoLayer this is negligible, but documents the pattern for larger apps.
    - Module loading: importing `effect` adds modules to the Deno module graph that must be loaded and initialized.

    Write these notes in clear, concise prose (not bullet-point template). Reference the actual numbers from the summary table. The goal is that someone reading RESULTS.md understands not just WHAT the overhead is, but WHY it exists and whether it matters for real workloads.
  </action>

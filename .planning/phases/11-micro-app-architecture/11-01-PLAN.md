Create `.planning/phases/11-micro-app-architecture/11-DECISION.md` by synthesizing the
research findings into a durable internal decision document. Use the structure below.

**Document structure (required sections):**

1. **Header** — Title, date, status (Accepted), one-line decision summary

2. **Problem Statement** — What mountApp failures were observed and under what
   conditions. Mention the three categories: islands not hydrating, routes not
   resolving, layout/context leaks. Note the intermittent nature and explain why
   (failures only appear when specific features are used).

3. **Root Cause Analysis** — Five specific root causes, each with:
   - Code location (file path + line range or function name)
   - What breaks
   - Trigger condition
   - The five are: (a) BuildCache singular/shared, (b) CommandType.App global
     leakage, (c) CommandType.NotFound global leakage, (d) ctx.state shared
     without namespacing, (e) _atomHydrationHook process-level global
   - Include the summary table from RESEARCH.md section 2 (Failure | Root Cause
     | Trigger)

4. **Options Evaluated** — Three options with equal treatment:
   - **Fix mountApp**: technical obstacles (4 items from research), gains,
     complexity (HIGH), verdict (high-effort, wrong layer for current use case)
   - **Programmatic Plugin Pattern**: how it works, why it avoids all mountApp
     failure modes, production evidence (workflowPlugin + authPlugin),
     limitations (no islands, no ctx.state namespacing), complexity (LOW),
     verdict (right choice)
   - **Module Federation**: why it's the wrong layer (server-side vs
     browser-side), build tool requirement (Vite, not esbuild), no Deno-native
     support, verdict (not applicable)
   - Include the comparison table from RESEARCH.md section 6

5. **Decision** — "Adopt the Programmatic Plugin Pattern as the supported
   composition model for Freak sub-apps." Include rationale bullets:
   - Production-validated by two real plugins (workflowPlugin, authPlugin)
   - Supports dynamic mount paths
   - Avoids all BuildCache/island/appWrapper failure modes
   - Low implementation effort (formalize what exists)

6. **mountApp Scope Clarification** — mountApp is NOT deprecated. Its valid
   scope: static prefix composition, no sub-app islands, no sub-app appWrapper,
   no fs-routes from sub-app. It is a utility for simple middleware-level
   composition. Document the specific conditions under which mountApp works
   correctly.

7. **Reference Implementations** — Point to the two canonical examples:
   - `workflows/src/plugin.ts` + `workflows/src/dashboard/app.ts`
     (workflowPlugin)
   - `authend/src/plugin.ts` + `authend/src/ui/app.ts` (authPlugin)
   - `platform/control-panel/main.ts` (host usage)
   - Describe the calling convention: `pluginName(config)(app)` — curried
     factory
   - Describe the layoutWrapper pattern for host shell integration

8. **Future Work** — Items for follow-on phases (NOT this phase):
   - Islands in plugins (requires BuildCache aggregation — non-trivial)
   - `Plugin<Config>` formal type in `@fresh/core`
   - ctx.state namespacing (typed sub-state per plugin)
   - Plugin authoring documentation and scaffolding

**Tone and format:**

- Internal notes, not a formal ADR — no "Status: Proposed/Accepted/Deprecated"
  ceremony beyond the header status line
- Specific and durable — someone reading this in 6 months should understand the
  decision and its basis without needing to re-read the research
- Reference specific code locations (file paths, function names) not vague
  descriptions
- Use markdown with headers, tables, and code blocks where they aid clarity
- Do NOT include the full mountApp source listing (that's in the research doc) —
  summarize the mechanism and cite the research
